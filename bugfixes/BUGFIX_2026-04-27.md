## 🚫 Remove the `(est)` weight marker (round 2 reversal)

### Bug 1 — `(est)` next to weights is noise — every weight is estimated

**Symptom.** Push Day workout card shows:
- `Barbell Bench Press — 4 × 12 — 215 lbs`
- `Overhead Press — 3 × 12 — 125 lbs (est)`
- `Close-Grip Bench Press — 3 × 12 — 165 lbs (est)`
- `Face Pull — 3 × 12 — 50 lbs (est)`
- `Tricep Kickback — 3 × 12 — 20 lbs (est)`

The `(est)` only shows up for some exercises (those without a direct PR on file). But every weight the app prescribes is a calculation — even the bench at 215 lbs is derived from a 1RM. Showing it for some and not others creates a false dichotomy: "these are estimates, but bench is somehow real." It isn't.

**Expected.** Drop the `(est)` text everywhere. All weights render the same way: just the number + unit.

**Acceptance criteria.**
- No `(est)` text anywhere in the workout card, live tracker, history, or sharing flow.
- The underlying `isEstimate: true` flag stays on the block (don't rip it out — it's useful for analytics and for any future feature that wants to surface "your bench prescription is based on direct PR vs your tricep kickback is derived from a multiplier").
- This is **purely a rendering removal**. Do not touch the conservative-weight calculation logic from round 2 — that part is correct and worth keeping. Just hide the indicator.

**Where to look.**
- `js/workouts.js` `_normalizeWeightDisplay()` — D2 added the `(est)` append here. Remove the append, leave the value formatting alone.
- Live tracker may have a parallel rendering path — grep for `isEstimate` and `(est)` across the codebase and remove every render-side usage.
- Acceptance: grep result for `"(est)"` in any non-test file should return zero matches after the fix.

## ⏱️ Today total doesn't match the workout cards

### Bug 2 — Home page says "2h 25m" but the two visible sessions sum to 1h 55m

**Symptom.** Home → Today section shows total duration **2h 25m** with two workouts visible: CSS Swim (60 min) and Push Day (55 min). 60 + 55 = 115 min = 1h 55m. The total is 30 min higher than the sum of the cards.

**Root cause (high confidence).** Two different duration formulas exist for strength workouts, and they don't agree:

1. **Workout-card formula (in `js/strength-workout-generator.js:148`):**
   ```js
   const estimated_duration_min = 10 + compoundCount * 12 + accessoryCount * 5;
   ```
   This is what produces the "55 min" badge on the Push Day card. After D2 (round 2 Bug 7), this was extended to fit-to-target so a requested 45 min lands closer to 45.

2. **Day-total formula (in `js/calendar.js:3013`, `_estimateStrengthSessionMin(exercises)`):**
   ```js
   const WORK_SEC = 45, SETUP_SEC = 120, WARMUP_SET_BOOST = 1.5;
   // per exercise: (sets + 1.5) × 45s + (sets + 0.5) × restSec(default 90s) + 120s setup
   ```
   This re-estimates from sets × reps × rest. For Push Day (1 × 4-set + 4 × 3-set), it produces ~57 min. Plus 60 min swim = ~117 min, but the screenshot says 145 min — so there may also be a warmup/cooldown buffer being added on top. Either way, **two formulas, two answers, never going to match.**

**Expected.** The total at the top of Today equals the literal sum of the workout cards' minute badges. Period.

**Acceptance criteria.**
- Day total uses each workout's `estimated_duration_min` field directly, not a re-estimate.
- For strength: read `w.estimated_duration_min` (set by the generator). If it's missing on a legacy/imported workout, fall back to `_estimateStrengthSessionMin(w.exercises)` so old data doesn't break.
- For cardio: read `duration` field directly (already works).
- Result: 60 + 55 = 1h 55m exactly.
- Same fix on the day-detail view (where you tap a date and see all sessions for that day) — the inline `day-totals-inline` chip should also reconcile.

**Where to look.**
- `js/calendar.js:3115–3133` — the `data.scheduledWorkouts.forEach()` loop that adds strength to `totalMin`. Replace the `_estimateStrengthSessionMin(w.exercises)` fallback with `w.estimated_duration_min || _estimateStrengthSessionMin(w.exercises)`.
- `js/calendar.js:3132` and `:3182` — both call sites need the same fix.
- `js/calendar.js:3013` — leave `_estimateStrengthSessionMin` in place as the legacy fallback, but add a comment that says *"Prefer w.estimated_duration_min (set by generator); this is the legacy fallback only."*
- After fix, manually verify: open today, confirm the chip at the top equals the literal sum of card badges. Open a day with 3+ sessions and verify same.

## 🧠 Strength generator doesn't know what you did 2 days ago

### Bug 3 — Push day repeats Overhead Press 2 days after a Saturday arms+shoulders session

**Symptom.** Saturday: I added an arms-and-shoulders session that included Overhead Press. Monday's auto-generated Push Day put Overhead Press right back on the menu. Two heavy OHP sessions 48 hours apart isn't great recovery and isn't what an intelligent planner would do.

**Expected.** When generating a strength session, the generator looks at the last 3 days of completed/logged strength work and **avoids repeating the same compound lift**. If the slot would have called for OHP, swap to a same-pattern alternative (Incline DB Press, Landmine Press, Push Press at lighter load).

**Acceptance criteria.**

For the generator:
- New input: `recentStrengthHistory: { lift, date, sets }[]` covering the last 3 days.
- For every compound lift the generator wants to schedule, check `recentStrengthHistory` — if the same lift was done within the last 48 hours, substitute from a same-pattern alternates list.
- If no clean alternate exists for the muscle pattern (e.g. you did everything that targets vertical pressing recently), drop the slot and add an extra accessory of a different pattern instead — don't force a duplicate.

For the caller (likely in `onboarding-v2.js` or the day-of-plan-rehydration code):
- Read recent completed strength sessions from localStorage / Supabase before invoking the generator.
- Only count *completed* sessions (not just planned), so a planned but skipped OHP doesn't artificially block today's slot.
- 3-day window is configurable via constant (`RECENT_LIFT_WINDOW_DAYS = 3`).

**Substitution table (push day examples):**
| Repeated lift | Same-pattern alternates |
|---|---|
| Barbell Bench Press | Incline Barbell Bench, DB Bench, Floor Press |
| Overhead Press | Incline DB Press, Landmine Press, Push Press (lighter), Z Press |
| Close-Grip Bench | Tricep-focused: Skull Crushers, JM Press, Dips |
| Squat | Front Squat, Bulgarian Split Squat, Hack Squat |
| Deadlift | Romanian Deadlift, Trap Bar DL, Sumo DL |

If the codebase already has a variant library (`window.VARIANT_LIBRARY_STRENGTH` is referenced in `strength-workout-generator.js:104`), reuse it — it likely already has the alternates organized by muscle pattern.

**Acceptance test (regression):**
- Seed a completion record: `{ exercises: [{ name: "Overhead Press", date: "2 days ago" }] }`.
- Call generator for today's push day.
- Assert no exercise in the output has name === "Overhead Press" — the slot should be filled by Incline DB Press or Landmine Press.

**Where to look.**
- `js/strength-workout-generator.js`:
  - `generateStrengthWorkout(opts)` line 99 — add `recentStrengthHistory` to opts.
  - The compound-lift loop at lines 109–116 — this is where the substitution check goes.
  - The accessories loop at lines 118–141 — same logic.
  - Use `window.VARIANT_LIBRARY_STRENGTH.variants[category]` to find alternates.
- Caller(s) of `generateStrengthWorkout`:
  - Find them: `grep -rn "generateStrengthWorkout(" js/`
  - Each caller needs to pass `recentStrengthHistory`. Build it from completed workouts in localStorage (the `completedWorkouts` or equivalent key — `js/calendar.js:2114` references the source of truth comment).
- For the "ignored my Saturday" specific case: the user-added session via Quick Add or Custom Plan should also flow into completion history when marked complete. Confirm those paths write to the same store the generator reads from.
