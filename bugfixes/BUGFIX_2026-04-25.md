## 🏃 Validation (soft warnings, not hard blocks)

### Bug 1 — Long Run cap should warn, not fully block

**Symptom.** Trying to manually add a Long Run when one already exists in the same week shows *"Can't add this session — Long Run is capped at 1 per week, full stop. You already have a Long Run on 2026-04-24."* The user is locked out entirely with no override.

**Why this matters.** Auto-generated plans should respect 1 long run/week. But manual additions are the user explicitly knowing what they're doing — the app shouldn't treat them like the planner. Sometimes life happens, you do two long runs, the app should let you and let the philosophy prompt you.

**Expected.** Show the warning with strong language ("Two long runs in a week is hard on recovery — you sure?") and a confirm button to override. Highlighting good, blocking bad.

**Acceptance criteria.**
- Long Run cap is a **warning** (severity: `warn`), not a **hard block** (severity: `block`).
- The dialog has two buttons: `Add Anyway` (primary) and `Cancel`.
- Override is logged (analytics event `long_run_cap_overridden`) so we can see how often this happens.
- The same softening applies to any other "full stop" hard block in the validator unless there's a real safety reason (e.g. injury return protocol).

**Where to look.**
- `js/workout-validator.js:59` — `rule_long_run_cap(ctx)`. Currently returns `severity: "block"`. Change to `severity: "warn"` and update the message.
- `js/add-running-session-flow.js:213` — duplicate fallback path with same hardcoded block. Same change.
- `js/session-type-library.js:290` — message string `"Long Run frequency is capped at 1 per week, full stop."` — soften to `"Two long runs in a week is hard on recovery."`
- Audit other rules in `workout-validator.js` for `severity: "block"` — flag any others where the user-knows-best argument applies.

## 💪 Strength Generator

### Bug 2 — Strength workouts pick weights without PR references; new PRs don't propagate

**Symptom (two halves).** 
1. Got a planned Overhead Press at **185 lbs × 8** when no OHP PR was on file. That's heavy for an 8-rep set even with a 275 bench. The generator made an aggressive assumption with no data.
2. Added a 205 OHP PR after the fact. The planned workout did not refresh — still showed 185.

**Expected.**
1. When no PR exists for an exercise, the generator should pick a **conservative** starting weight and label it as "Estimate" (not a real recommendation).
2. When a PR is added/updated, **today's and all future** planned strength sessions immediately recompute their weights.

**Acceptance criteria.**

For #1 (no-PR indicator + conservative defaults):
- Each strength exercise in the generator output gets an `isEstimate: true` flag when the weight was derived without a PR reference.
- UI renders a small `~` or `(est)` next to estimated weights with a tooltip: *"No PR on file — start light and adjust."*
- Default formula when no PR: 
  - Compound lifts (squat/bench/DL/OHP): `0.5 × bodyweight × rep_factor` where `rep_factor = 0.75` for 1RM, decreasing with rep count (8 reps → ~0.6 of 1RM).
  - Accessory lifts: 50% of the parent compound's estimated weight.
  - All estimates rounded down to nearest 5 lbs (conservative bias).
- Bench press 275 / OHP unknown → estimated OHP-for-8 should land at ~95–115 lbs, not 185.

For #2 (PR propagation):
- Adding/updating a PR via Training → Training Zones & Strength Benchmarks → Update Lifts triggers `recomputePlannedStrengthSessions()`.
- That function walks `workoutSchedule`, finds entries with `type === "weightlifting"` AND `date >= today`, and re-runs the weight-population step using the new PRs.
- Existing user-edited weights are NOT overwritten — only weights that came from the generator (flagged `weightSource: "generated"`) get refreshed.
- Toast confirmation: *"Updated 4 upcoming sessions with your new max."*

**Where to look.**
- `js/strength-workout-generator.js`:
  - `estimated_duration_min` computation at line 148 — but more importantly, find where each block's `weight` field gets populated. Search for `weight:` in the block-building loops.
  - The block construction loop is around lines 110–145 — that's where the `isEstimate` flag should be added.
- `js/workouts.js` `_normalizeWeightDisplay()` — touchpoint for rendering the `(est)` indicator. Update this to read `isEstimate` and append the marker.
- For PR change → recompute hook: find where `Update Lifts` saves to localStorage. Add a follow-up call to a new `recomputePlannedStrengthSessions(profileChanges)` function. Likely in `js/workouts.js` near `buildLiftingDay` or in a strength-benchmarks UI handler — grep for `Update Lifts` in `index.html` and `js/`.
- `js/calendar.js` — `renderCalendar()` should be invoked after recompute so the user sees the change immediately.

### Bug 7 — "45-min" workout came in at 41:30

**Symptom.** Asked for a 45-minute weightlifting session, got 6 exercises summing to ~41.5 minutes. Off by a set or two of one exercise.

**Expected.** Generator targets the requested duration within ±2 minutes. If the current exercise selection comes in short, bump one exercise from 3→4 sets (or add a small accessory) until it fits.

**Acceptance criteria.**
- After exercise selection, generator computes `estimated_duration_min`. If `targetDuration - estimated > 2`, generator adds work (extra set on a major compound, or one bodyweight finisher) until `|target - estimated| ≤ 2`.
- If estimate exceeds target by >2, generator drops the lowest-priority accessory.
- Final estimate is logged in workout metadata so the calendar card displays the actual estimate (45 min, not the requested 45).

**Where to look.**
- `js/strength-workout-generator.js:148` — current formula `10 + compoundCount × 12 + accessoryCount × 5`. After computing initial estimate, add a fit-to-target loop that adjusts set counts.
- The function returns `workout.all_blocks` — adjusting `block.sets` is how to bump volume without restructuring.
- Test case: request 45 min for a push day → confirm output lands in [43, 47].

### Bug 8 — Circuit Add Exercise modal: keyboard covers the input

**Symptom.** Tapping the Exercise Name field in the circuit Add Exercise modal opens the iOS keyboard, which rises and covers the input. User can't see what they're typing.

**Expected.** When the keyboard opens, the input scrolls (or the modal slides) so the input is above the keyboard with a few px of padding.

**Acceptance criteria.**
- Input remains visible above the keyboard on iPhone (test on 14 and SE since SE has the smallest viewport).
- No horizontal layout breakage during keyboard transition.
- Works whether the user opens the modal scrolled to top, middle, or bottom of the parent screen.

**Where to look.**
- `js/ui/circuit-builder.js:667` — `Add Exercise` modal HTML. The modal class is `circuit-modal-overlay` / `circuit-modal`.
- `style.css` — find `.circuit-modal-overlay` and `.circuit-modal`. The fix is one of:
  1. **Recommended:** Add `position: fixed` on the modal + use `visualViewport` API to recompute modal `top` when keyboard opens.
  2. Add an `onfocus` handler to the input that calls `el.scrollIntoView({ block: "center" })`.
  3. Set `padding-bottom: 50vh` on the modal body during input focus so the input has room to scroll up.
- iOS Capacitor specific: there's a `Keyboard` plugin (`@capacitor/keyboard`) that emits `keyboardWillShow` / `keyboardWillHide` events. Listening to those is the cleanest fix on native — push the modal up by the reported keyboard height. Check `capacitor.config.ts` for whether the plugin is installed.

## 🏋️ Hyrox

### Bug 9 — Hyrox needs a 5th focus type: Mixed Intervals (run + stations)

**Symptom.** Current Hyrox auto-generate offers 4 focus types: Easy Aerobic / Run Intervals / Stations / Race Simulation. None capture the bread-and-butter Hyrox session: short run intervals interleaved with 1–2 stations between them. That's how Hyrox is actually raced and how most Hyrox athletes train.

**Research note.** Real Hyrox programming includes:
- **Easy aerobic base** — Z2 running, like marathon training. ✓ already have.
- **Pure run intervals (track work)** — speed and threshold. ✓ already have.
- **Skill / station-only days** — practicing technique under fatigue without runs. ✓ already have.
- **Race simulation** — full or half Hyrox at race pace. ✓ already have.
- **Mixed intervals** — 4–8 rounds of `1K run + 1–2 stations`, the highest-frequency session in most Hyrox training programs. **MISSING.**

The technical name for running-while-fatigued-from-stations is "compromised running" — accurate but sounds bad in a UX. Use **"Mixed Intervals"**.

**Expected.** Add Mixed Intervals as the 5th focus, listed second (it's the most common session, should be high in the dropdown).

**Acceptance criteria.**
- Dropdown order: Easy Aerobic → **Mixed Intervals** → Run Intervals → Stations → Race Simulation.
- Mixed Intervals subtitle: *"Run intervals broken up by 1–2 stations between"*.
- Generator produces sessions like:
  - **Beginner:** 4 rounds of `800m run + 50m sled push (light)`.
  - **Intermediate:** 6 rounds of `1K run + 50m sled push + 50m sled pull`.
  - **Advanced:** 8 rounds of `1K run + 1 station rotating through all 8`.
- Stations rotate across rounds — same station shouldn't appear back-to-back.
- Estimated duration: 45–75 min depending on level.

**Where to look.**
- The Hyrox quick-add picker shipped in C7c (commit `2a21354`). Find the focus-type array and add the new entry.
- `js/admin-workouts.js:27` — `hyrox: ["easy", "intervals", "stations", "race_simulation"]` — add `"mixed_intervals"` to this list.
- The 8 Hyrox stations (SkiErg, Sled Push, Sled Pull, Burpee Broad Jump, Rowing, Farmer's Carry, Sandbag Lunges, Wall Balls) are already wired — reuse from the Race Simulation generator and just sub in shorter run distances + fewer stations per round.

## 🔴 Live Workout Tracker — P0 BUNDLE

These three are bundled because they share the same state machine. **Data-loss risk on #4 and #5.** Ship together.

### Bug 4 — Finish button stops working after dismissing the save modal — P0

**Symptom.** During a live workout: tapped **Finish** prematurely → save-choice modal appeared (`Log Everything as Planned` / `Save What I Completed` / `Keep Going`) → user closed the modal (X or back) → tapped Finish again → **nothing happens**.

**Repro.**
1. Start a strength workout.
2. Log the first 1–2 sets.
3. Tap Finish.
4. Modal appears.
5. Dismiss the modal (any non-button path: tap outside, hardware back, X if rendered).
6. Tap Finish again.
7. Nothing happens.

**Expected.** Finish reliably opens the save modal every time it's tapped. Modal dismiss leaves the tracker in a clean state.

**Likely cause.** Looking at `js/live-tracker.js:1113` `_showFinishChoiceModal()`:
- Comment in code says *"No tap-outside dismiss. Previously tapping the dim backdrop silently closed the modal without saving"* — so the dev tried to prevent this, but the modal can still be dismissed via other paths (browser back, system back gesture, swipe-down on iOS).
- There's no event listener on the modal's removal that re-arms the Finish button — so once a dismiss happens through any path other than the explicit buttons, Finish is now wired to a stale handler or `_liveTracker` is in a partial state.

**Acceptance criteria.**
- Finish button is functional after every modal dismiss path (tap outside, X, system back, swipe down).
- Multiple consecutive Finish taps are idempotent — they always re-open the modal, never error or hang.
- The save modal's open/close should be a pure UI overlay — `_liveTracker` state should not change until the user taps `Log Everything` / `Save What I Completed` / commits via Hyrox/Endurance modals.
- Add a `data-state="finishing"` flag on the tracker; clear it on modal dismiss; only `_finishLiveWorkout()` checks the flag to prevent stacking duplicate modals.

**Where to look.**
- `js/live-tracker.js:979` `_finishLiveWorkout()` — entry point.
- `js/live-tracker.js:1113` `_showFinishChoiceModal()` — the modal that's getting dismissed.
- `js/live-tracker.js:533` and lines 719/752 — the three places `_finishLiveWorkout()` is called from. Confirm the click handler isn't being unbound.
- Add a defensive cleanup: at the top of `_finishLiveWorkout`, if a `live-finish-choice` overlay exists from a previous attempt, remove it before creating a new one.

**Expected.** Mark as Complete should produce the same final state as a clean live-tracker save: completion record written, Strava upload triggered (if configured), feedback captured, hydration/nutrition recalc fired (Bug 10).

**Acceptance criteria.**
- The Mark as Complete flow calls the same finalization function as the live tracker's `_commitLiveWorkout`. Refactor: extract the post-commit pipeline (Strava push, feedback prompt, hydration/nutrition recalc, calendar refresh, training-block update) into a single function `finalizeWorkoutCompletion(record)` that both paths invoke.
- If Strava is configured and the user is connected, Mark as Complete pushes to Strava.
- Strava push failure is non-blocking but logged + surfaced as a small toast: *"Saved locally — Strava push failed, will retry."*

**Where to look.**
- `js/live-tracker.js:_commitLiveWorkout` (around line 1140) — current "good" save path.
- `js/calendar.js:2560` — `toggleCompletionForm()` — the Mark as Complete entry. Trace what it actually invokes.
- `js/strava-integration.js` — `pushToStrava()` or similar. Confirm it's only called from `_commitLiveWorkout`.
- `js/feedback-tracker.js` — the "How did that feel?" prompt. Both paths should invoke it.

## 💧 Hydration + Nutrition

### Bug 10 — Targets don't recalc when a workout is logged as completed

**Symptom.** Manually logged a 10-mile run as completed. Hydration target stayed at *"133oz — 117oz base + 16oz for your biceps, triceps, shoulders — 45 min"* — still pegged to the morning's planned strength session. The 10-mile run added zero ounces to the target.

**Expected.** When a workout is marked complete, hydration and nutrition targets recompute immediately based on the *actual* completed work, not the planned work.

**Why this matters.** The planned target is a forecast; the completed target is reality. A user who plans 45 min lifting but actually does 100 min running needs ~50% more water and ~600 more carbs. The current behavior leaves them under-fueled.

**Acceptance criteria.**
- `getHydrationBreakdownForDate(dateStr)` prefers completed workouts over planned ones when both exist for the same date. Completed workouts contribute their actual `durationMin` and `type`.
- If multiple workouts on the same day (e.g. AM run + PM lift, both completed): both contribute their bonuses (additive).
- Calorie target similarly increments based on the completed workout's training-day adjustment (`trainingDayAdjustmentText` in `nutrition-calculator.js` line 661).
- Nutrition recompute call happens via the new `finalizeWorkoutCompletion()` from Bug 5.
- UI updates without page refresh: hydration card and "Nutrition Targets Today" card both repaint.
- Sanity check: target shouldn't decrease if the planned work was *more* than the actual (don't penalize someone who logged a partial workout).
  - Implementation: `effectiveBonusOz = max(plannedBonus, completedBonus)`. Same for calories.

**Where to look.**
- `js/hydration.js`:
  - `getWorkoutInfoForDate(dateStr)` line 440 — currently reads only from `workoutSchedule` (planned). Update to also pull from completion records (`completedWorkouts` localStorage key, or whatever calendar.js uses — see `js/calendar.js:2114` comment "what the Mark as Complete flow sets").
  - `computeWorkoutBonusOz(type, durationMin, opts)` line 92 — reuse this for completed workouts.
  - `getHydrationBreakdownForDate(dateStr)` line 538 — top-level integrator.
- `js/nutrition-calculator.js`:
  - `calculate(classification, profile)` line 670+ — currently doesn't read today's completed sessions. Add a `todayActiveMinutes` derivation from completion records and bump the target accordingly.
- `js/calendar.js:2114–2123` — comments reference both "what the Mark as Complete flow sets" and Strava-imported workouts as completion sources of truth. Read both.
- After recompute: call `renderHydrationTransparency()` (in hydration.js) and `updateNutritionDashboard()` (in nutrition-v2.js) to repaint without reload.
